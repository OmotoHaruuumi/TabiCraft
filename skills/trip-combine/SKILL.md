# trip-combine — 組み合わせ案の大量生成（Phase 2）

「行きたいことリスト」を、実際に成立する旅程の**組み合わせ案**へ変換する。1案に絞らず、多数の案を並べて選ばせる。
出力は旅行フォルダの `combinations.md`。

## 入力
- `trip-basics.md`（無ければ trip-basics を先に案内）
- `wishlist.md`（無ければ、共有マスター `~/.claude/travel/templates/wishlist-template.md` を旅行フォルダに `wishlist.md` としてコピーし、「自分で書く / 対話で埋める」の2択を提示。対話を選んだら AskUserQuestion で聞き取りこちらが記入する）
- `~/.claude/travel/profiles/*.md`（同行者ぶん）

## 入力モード（最初に確認）
着手前に AskUserQuestion で**案を誰が出すか**を聞く（決めるのは常にユーザー）：**自分で案を出す / Claudeに案を出してもらう / Claudeと一緒に案を出す（既定） / 一度止めて相談する**。
**「一度止めて相談する」は、このスキルで選択肢を出すすべての場面で必ず付ける**（選ばれたら自動進行を止め会話する）。
- 自分で案を出す：ユーザーが組み合わせの骨子（行く順・大枠）を挙げる → Claudeは成立性チェック・地理整理・リンク補助に徹する。
- Claudeに案を出してもらう：wishlist＋プロファイルから Claude が案を多数生成（既定の挙動）。
- wishlist 自体も「自分で書く / 対話で埋める」を選べる（入力の項参照）。

## 手順

### 0. 着手前チェック：外せないもの（マスト）— 生成を始める直前に必ず
案の生成（STEP 3）に入る**直前**に、wishlist の「外せないもの（マスト）」欄を Read で確認する。
**空（未記入）なら、生成を始めずに** AskUserQuestion で選ばせる:
- **今ここで外せないものを書く** … AskUserQuestion で聞き取り、こちらが `wishlist.md` の欄に追記してから生成へ。
- **書かずにこのまま進む** … マスト指定なしとして生成へ。
- **一度止めて相談する** … 自動進行を止めて会話する。
記入済み／追記されたマスト項目は、**できる限り全案に含める**。日数・距離の都合で物理的に入れ切れない場合は、無理に詰め込まず **要注意「このマストは入れられませんでした（理由）」を目立つように明記**し、代替（日程延長・他を削る・近い別案など）を提案する。

### 1. 読み込み・整理
- wishlist を Read し、4カテゴリに仕分け: 行きたい場所/やりたいこと・宿の条件・食べたいもの・備考。
- 外せないもの（マスト）を最優先制約として把握する。
- trip-basics の泊数・到着帰宅パターン・季節の罠・予算を制約として把握。
- プロファイルから各人の雰囲気・ペース・苦手を把握。

#### 1.5 宿の決め方を確認（宿固定派 / エリア先決め派）
宿の扱いは人によって違うので、案づくりの前に AskUserQuestion で聞く（「一度止めて相談する」も必ず含める）:
- **宿そのものにこだわる（宿固定派）** … 具体の宿名を先に決め、旅程はその宿の立地に合わせて組む。候補があれば**ハード条件（譲れない設備・予算・苦手＝虫等）と1件ずつ照合**し、**条件と候補の矛盾を早期に明示**する（例「プライベートプール希望だが候補は共用プールのみ」）。
- **泊まるエリアを先に決めたい（エリア先決め派）** … まず拠点エリア（と宿替えの有無）を決め、宿は後で。各エリアにつき**2択程度の候補を名前で提示**し、最終決定は予約時でOK（この場合 brochure で「宿2択」レイアウトに繋げる）。
- wishlist に宿の条件・候補名があるかで初期推測してよいが、**決めつけず必ず確認**する。
- どちらでも、宿は trip-basics/refine と矛盾させない（決めた宿名・エリアは以降ブレさせない）。
- **`spots.md`（スポット調査表）があれば Read** し、調査済みのスポットを把握（同じものを二度調べない）。

### 2. リサーチ＆スポット調査表（調べたら必ず表に記録）
足りない情報を WebSearch で補う。多い場合は Agent を並列で手分け（観光/グルメ/移動・宿/治安季節 の4観点）。
**一度調べたスポットは必ず `spots.md` の表に1行追記する**（先に Read 済みの行は再調査しない・追記専用）。各行に次を記録:

| スポット | エリア・方角 | 所要時間目安 | 金額(目安) | アクセス | 備考 |
|---|---|---|---|---|---|

- **エリア・方角**：街の中心から見た位置＋エリア名（例「中心部」「北東・モンティ地区」「南西・トラステヴェレ」）。地理クラスタリングに使う。
- **所要時間目安**：見学・滞在の目安（例: 1.5h）。
- **金額(目安)**：入場料・体験料など。**信頼度を付す**（実査 / 参考~ / 推定≈）。無料は「無料」。通貨は現地通貨＋日本円換算。
- **アクセス**：最寄駅・バス停／そこから徒歩何分／主な手段。
- **備考**：予約要否・定休日・混雑時間帯・服装規定・ベスト時間帯など。

調べ終えたら**この表をユーザーにも提示**する。表は trip-refine でもそのまま使い回す（再調査ゼロ）。
- 宿候補のエリアと価格帯、wishlist の食を出せる店・エリアも同様に調べ、必要なら表に含める。

### 3. 組み合わせ生成（核心）
**地理クラスタリング＋ペース＋到着帰宅パターン**で、性格の異なる案を多数作る。最低5案、できれば7〜8案。
案ごとに軸を変える。例:
- 王道全部入り / 一点豪華（1テーマ深掘り） / のんびり連泊 / 欲張り周遊 / グルメ重視 / 自然重視 / 雨でも楽しめる代替版
- 到着帰宅パターンが複数あるなら、主要パターン別に初日・最終日の使い方を変えた版も作る。

**マスト項目はできる限り全案に入れる**（軸を変えても優先的に確保）。落とすのはまずマスト以外から。
物理的に入れ切れない案では無理に詰め込まず、**要注意「入れられなかったマストと理由」を目立つように明示**する（例: 「要注意 ◯◯は距離的に5泊では不可。+1泊か△△を諦めれば可能」）。
各案には必ず:
- **キャッチコピー**（その案の性格が一言で分かる）
- **ざっくり日割り**（Day1〜の見出しだけ。詳細は refine フェーズ）
- **wishlist の何が入って何が落ちたか**（取捨選択の明示。マストは原則入る／入らない場合は要注意で理由を明記）
- **想定予算レンジ**と**こんな人向け**

#### 3.5 動線サニティ（往復・二度手間を自動でフラグ）— 各案に必ず適用
案を作ったら、**移動を必ず地理で点検**してから出す（ユーザーに指摘される前に自分で潰す）。
- **エリアでクラスタリング**：spots.md の「エリア・方角」を使い、同方面のスポットは同日にまとめる。1日の中で東↔西↔東のような**ジグザグ**、別々の日に同じ遠方へ**2回行く二度手間**を検出する。
- **見つけたら警告して直す**：「この案は2/22にA(西)→B(北)→A(西)の大往復」等を明示し、**並び替え/日の入替/拠点変更**で往復を消した改善版にする。
- **発着の寄せ**：**初日は到着地点（空港・駅）の近くから**始め、**最終日は帰路方向（空港・駅）に寄せて締める**（最終日のアクティビティは帰り道の途中に置けないか必ず検討）。
- **宿替えがある案**は移動の所要・荷物の扱い（朝イチ移動→預けてから遊ぶ等）も一言添える。

### 4. 気づき提案（プロファイル活用・重要）
wishlist に無いが**プロファイルから合いそうなもの**を2〜4件、「提案」として別枠で出す。
- 根拠を必ず添える: 「〇〇さんはローカル屋台好きと学習済み → △△の夜市もおすすめ」。
- 採否はユーザーに委ねる（押し付けない）。
- ここで新たな好み発言が出たら trip-profile モードB で観察ログに追記。

### 5. 保存
- 調査表は `spots.md` に保存／追記（次回・refine で再利用。再調査を防ぐ）。
- `combinations.md` を Write。構成:
```
# 組み合わせ案：<旅行名>
## 前提（泊数・到着帰宅・予算・メンバー要約）
## 主要スポットの簡単ガイド（地名・固有名詞を知らない人でも読めるよう、各地名を1行で説明）
## 案A〈キャッチコピー〉   … 日割り見出し / 入った要素・落ちた要素 / 予算 / 向き
## 案B 〜
## プロファイルからの追加提案（根拠つき）
## 比較表（案 × マスト充足・雰囲気・予算・移動量・体力・天気耐性）※マスト未充足は要注意で明示
```
- **前半に必ず「主要スポットの簡単ガイド」を入れる**：案の中に出てくる地名・エリア・スポット・固有名詞（例: パトン/オールドタウン/各島/施設名/港/配車アプリ等）を、**その土地に詳しくない同行者でも分かるよう1行ずつ平易に説明**する。エリア（泊まる場所）／島・海／遊び・観光／港・足 のように小分類すると読みやすい。案を追加・変更したら、新しく登場した固有名詞もこのガイドに追記して同期させる。
- 末尾に**比較表**を付け、横断で選びやすくする。

### 6. 次フェーズへ案内
> 組み合わせ案を `combinations.md` に作りました。比較表も付けています。
> 気になる案を選んでください。選んだ案を個別ファイルに展開して、一緒に詰めていきます（`trip-refine`）。

## 注意
- 案は多様性が命。似た案を量産しない（軸を必ず変える）。
- ここで分単位の時刻は決めない（refine の仕事）。日割り見出しレベルに留める。
- **`combinations.md` には結果（最新の案）だけを載せる。変更履歴・改訂理由・「旧版はこうだった」等のメモは書かない**。案を直すときは古い記述を上書き/削除し、常に「今の確定形」だけが残る状態にする（変更の経緯はチャットで説明すれば十分）。「Fix-1に改訂」「○○を解消」などの注記も残さない。
