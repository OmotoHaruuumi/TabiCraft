# trip-brochure — 旅のしおり HTML 生成（Phase 4）

確定プランを、見て楽しく実際に使える**単一ファイルHTMLのしおり**にする。
出力は旅行フォルダの `output/<旅行名>しおり.html`（CSS/JSは内部化・1ファイル完結）。

## 入力
- `plans/<案名>.md`（確定版。複数あればどれか確認）
- `trip-basics.md`（前提・通貨・季節メモ）
- 参考テンプレート: スキルフォルダ内 `assets/brochure-template.html`

## 入力モード（最初に確認）
着手前に AskUserQuestion でデザインの決め方を聞く：**自分で指定（配色・載せる情報・トーンを指示） / おまかせ（行き先の雰囲気でこちらが最適化） / 一度止めて相談する**。
**「一度止めて相談する」は、このスキルで選択肢を出すすべての場面で必ず付ける**（選ばれたら自動進行を止め会話する）。
- 自分で指定なら、その希望（色・順序・載せる/省く項目）を反映。
- おまかせなら行き先の雰囲気で配色・装飾を自動決定。

## 手順

### 1. データ抽出
plan md と basics から、しおりに載せる要素を構造化して取り出す:
- 旅行名・期間・メンバー・ルート / 宿（都市ごと。固定 or エリア2択）/ Day別タイムライン / 移動手段 / 予算内訳 / **予約・手配チェックリスト** / 持ち物・準備 / 現地メモ（気候・通貨・注意）

### 2. Google Maps 経路リンク生成（各移動行）
形式:
`https://www.google.com/maps/dir/?api=1&origin=<FROM>&destination=<TO>&travelmode=<MODE>`
- travelmode: transit（電車地下鉄バス・空港送迎）/ walking（徒歩15分以内）/ driving（郊外）/ bicycling（広い公園史跡）
- 場所名は英語で（例: `Wat+Pho+Bangkok`）。
- **各日の朝の origin は必ず宿名**（エリア名・駅名はNG）。例: `origin=Riva+Surya+Bangkok`。
  - **例外**：宿が未確定（エリア先決め派で「宿2択」を残す場合）は、宿名の代わりに**エリア名/最寄りランドマーク**を origin に使ってよい（例 `origin=Phuket+Old+Town`）。確定したら宿名に差し替える。
- 認証不要のただのURL。外部連携ではないので「HTMLのみ」方針と矛盾しない。不要ならこのステップを省ける。

### 3. HTML 生成（絵文字は使わない・画像スロットで魅せる）
`assets/brochure-template.html` を土台に、抽出データを流し込む。品質要件:
- **単一ファイル**（CSS/JS内部化）。**絵文字は一切使わない**。装飾は配色・余白・画像で表現する。
- **画像スロットを置く**：ヒーローに `images/hero.jpg`、各日の先頭に `images/day1.jpg`…`dayN.jpg`。
  テンプレの `.imgslot` 機構により、画像が無い間はパス入りのプレースホルダーが表示され、画像を入れると自動で写真に切り替わる。
- **画像サイズの目安**：横は**コンテナ幅いっぱい（width:100%）**、高さは **ヒーロー≒220px／各日の写真≒300px** を基準にしっかり見せる（`.day .dayimg{height:300px}` 目安。`object-fit:cover` で中央トリミング）。各日の写真が小さいと埋もれるので、テンプレ既定値が小さい場合は**この高さまで上げる**。大きすぎ/小さすぎはユーザーの好みで微調整可。
- 各スポットのタイトルは**絵文字なしのプレーンテキスト**。アイコンの代わりに画像・配色・時間帯ラベルで構造を出す。
- **レスポンシブ**（スマホで旅行中に見る前提。タップしやすいボタン）。
- **行き先の雰囲気で配色を変える**（南国＝暖色、欧州都市＝落ち着いた色 など。毎回同じにしない。ただし絵文字は足さない）。
- 構成: ヒーロー（画像＋旅行名・日付・ルート）→ 概要（宿/通貨/移動アプリ/注意）→ 予算→ Day別（日ごとに画像スロット＋タイムライン＋経路ボタン）→ **予約・手配チェックリスト**→ 持ち物→ 現地メモ。
- **plan の「予約・手配チェックリスト」を必ず載せる**（タスク/締切/順番。出発前のやることを同行者と共有できるように）。
- **用語は必ずどこかで説明**：案・しおりに出す**固有名詞/地名/施設名（例: バングラ通り 等）は、初出時か用語ガイドで1行説明**を付け、知らない同行者でも読めるようにする（取りこぼしを生成後にチェック）。
- 予算は信頼度（実査/参考~/推定≈）を視覚的に区別。日本語UI・温かいトーン。
- 生成後、タグ閉じ・JS構文をざっと自己チェック。

#### 宿2択レイアウト（エリア先決め派のとき）
combine/refine で「**エリア先決め派**（宿は最終未確定で各エリア2択を残す）」だった場合、宿を**チェックインのタイミングに2択として埋め込む**:
- その日の「ホテルへチェックイン」の行に、**選択肢1 / 選択肢2** を写真付きで横並び表示（旅程本体は共通、宿だけ選ぶ形）。
- 2択の下に「**その他の候補（名前のみ）**」を添える（あれば）。
- 宿が未確定なので、その日の経路リンク origin はエリア名でよい（上記「2.の例外」）。
- 「宿固定派」のときはこのレイアウトは使わず、確定した単一の宿名で書く。

### 3.5 画像フォルダの用意とユーザーへの案内（重要）
しおりは絵文字でなく**ユーザーの写真**で魅せる。差し込み場所とパスを用意し、フォルダに入れてもらう。
- しおりHTMLと同じ階層に **`images/` フォルダを作成**する（`output/images/`）。Bash で `mkdir -p`。
- **拡張子は決め打ちせず、実ファイルに合わせる**：ユーザーが画像を入れたら `output/images/` を Bash でスキャンし、`hero.*` / `dayN.*` / 宿画像などを**実在ファイル名（.jpg/.jpeg/.png/.webp 等）へ自動でマッピング**して `<img src>` を書き換える（今回 `.jpeg`/`.webp` のズレを手で直した反省）。例: `ls output/images` の結果と突き合わせて src を更新。
- HTML が参照する画像ファイル名の一覧を**ユーザーに伝え、そのフォルダに置いてもらう**。例:
  > 写真を入れるとしおりが映えます。`output/images/` フォルダに次の名前で画像を入れてください（任意・無くてもプレースホルダーのまま使えます）:
  > ・hero.jpg（表紙のメイン画像）  ・day1.jpg 〜 dayN.jpg（各日の写真）
- 画像は個人ファイル。`output/` 配下なので gitignore 済み（公開されない）。

### 4. 保存・共有
- `output/<旅行名>しおり.html` に Write し、`output/images/` フォルダを作成。
- パス・画像フォルダ・入れる画像名を伝える。
> 旅のしおりを `output/<旅行名>しおり.html` に作りました。ブラウザで開いて確認してください。
> 写真を入れるとさらに映えます。`output/images/` フォルダに hero.jpg / day1.jpg〜 を入れてください（任意）。
> 直したい所（配色・載せる情報・予定）があれば言ってください。経路リンクが不要なら外せます。

### 4.5 共有用に単一ファイル化（重要・スマホ共有の現実）
**素の `<旅行名>しおり.html` を単体で送ると、画像は別フォルダ参照のため写真が全部消える**（LINE/メールでHTMLは扱いづらい）。共有するなら必ず次のどちらかにする。AskUserQuestion で「PDFで保存 / 画像埋め込み単一HTML / 両方 / 一度止めて相談する」を提示。
- **PDF（スマホ共有の本命・最推奨）**：1ファイルに写真も内包・誰でも開ける・地図リンクもタップ可。
  - `~/.claude/skills/trip-brochure/export.sh output/<旅行名>しおり.html output/<旅行名>しおり.pdf`（Chrome/Chromium/Edgeヘッドレス。無ければHTML右上の「PDFで保存」ボタン＝`window.print()`を案内）
  - PNG も同様: `export.sh ... output/<旅行名>しおり.png`
- **画像埋め込み単一HTML**：レスポンシブのまま1ファイルで完結（相手はブラウザで開く）。`images/...` 参照を **base64データURIに埋め込む**。
  - `~/.claude/skills/trip-brochure/embed_images.py output/<旅行名>しおり.html output/<旅行名>しおり_単一ファイル.html`
- 画像PNG等が大きいと数MBになる。重ければ画像を横1200px程度に縮小すると軽くなる旨を案内。
- 出力先は `output/` 配下（gitignore 済みなので公開されない）。

### 5. 学習ループ（締め）
しおり完成は1旅行の区切り。気に入った点・反省点が会話に出たら trip-profile モードB で各人の観察ログに追記し、次回の提案精度を上げる。

## 任意の拡張（求められたら）
- Leaflet + CARTO の軽量インタラクティブ地図埋め込み（オフライン耐性・iframe不使用）。
- 印刷/PDF向けレイアウト。
- 多言語併記（現地語の指さしフレーズ）。

## 注意
- plan が未確定なら refine に戻す（確定前にしおり化しない）。
- 宿名・時刻は plan md と完全一致させる（勝手に変えない）。
- **成果物には最新の確定形だけを残す**（変更履歴・改訂理由・旧版メモは書かない。経緯はチャットで）。
- **共有する前に必ず単一ファイル化**（4.5）。素のHTML単体は画像が飛ぶので相手に渡さない。
