# trip-plan — 旅行計画の司令塔（オーケストレーター）

ユーザーは内部の5スキルを知らなくていい。気軽な一言（「沖縄に家族で行きたい」等）から、
**状態を自動判定 → 適切なフェーズへ誘導 → 完了したら次へ繋ぐ** を回す入口。

## 基本動作（毎回これを行う）

### STEP 1. 旅行フォルダを確立する
計画ファイルを置く「旅行フォルダ」を決める。
1. カレントディレクトリに `trip-basics.md` / `wishlist.md` / `combinations.md` / `plans/` のいずれかがあれば、**ここを旅行フォルダとみなす**（既存の計画の続き）。
2. 無ければ、ユーザーの発言から旅行名を推測し（例: タイ3泊→「タイ旅行」）、AskUserQuestion で確認:
   - 「この旅行の作業フォルダを作りますか？ 候補: `~/旅行/<旅行名>/`」
   - はい → Bash で `mkdir -p` し、以後そこを旅行フォルダとして使う（生成ファイルは全てここ）。
   - 既存フォルダを使う → パスを聞く。
3. 旅行フォルダのパスを把握し、以降の Read/Write/状態判定は**すべてこのフォルダ基準**で行う。

### STEP 2. 状態を自動判定して現在フェーズを決める
旅行フォルダ内のファイル有無を Glob/Read で調べ、下表で「今やるべきフェーズ」を判定する。

| 判定条件（上から順にチェック） | 現在地 | 起動するスキル |
|---|---|---|
| 同行者のプロファイルが `~/.claude/travel/profiles/` に無い、または「好みを覚えて/更新して」と言われた | Phase 0 | **trip-profile** |
| `trip-basics.md` が無い | Phase 1 | **trip-basics** |
| `trip-basics.md` はあるが `wishlist.md` または `combinations.md` が無い | Phase 2 | **trip-combine** |
| `combinations.md` はあるが、確定した `plans/<案名>.md` が無い | Phase 3 | **trip-refine** |
| 確定 plan はあるが `output/*.html` が無い | Phase 4 | **trip-brochure** |
| 全て揃っている | 完成 | 修正対応 or 新フェーズ提案 |

- 判定結果を**一言でユーザーに伝えてから**進む（例: 「基本情報がまだなので、まず行き先と日程から決めましょう」）。
- ユーザーが特定フェーズを明示（「しおりだけ作り直したい」）した場合は判定を飛ばしてそのフェーズへ。

### STEP 3. 該当スキルを実行する
判定したスキルを Skill ツールで呼び出す（例: `trip-basics`）。各スキルは自分の入力を Read し、出力を旅行フォルダに保存する。
- プロファイルが無い状態で basics/combine に来たら、先に trip-profile を軽く回すか、「プロファイルは後で作る」を選べるようにする（強制しない＝気軽さ優先）。

### STEP 4. フェーズを自動で繋ぐ
1フェーズ終わったら、止まらずに**次フェーズへ橋渡し**する:
> 基本情報がまとまりました。このまま行きたい所・食べたい物を集めて、プラン案を出しましょうか？
- ユーザーが「うん」「お願い」なら次フェーズ（STEP 2の判定を再実行）へ進む。
- 「今日はここまで」なら状態を保存して終了（次回 trip-plan を呼べば続きから再開できると伝える）。

## 気軽さのための原則
- **🛑 選択肢を出すときは必ず「一度止めて相談する（このまま会話で詰めたい）」という選択肢を含める**（全フェーズ・全 AskUserQuestion 共通の絶対ルール）。これが選ばれたら自動進行を止め、選択式をやめてユーザーと自由に会話する。エージェントが勝手に先へ進んでユーザーを置いていかない。
- **専門用語・スキル名を表に出さない**。ユーザーには「次は〇〇しましょう」と日本語で言う。内部でどのスキルを使うかは見せない。
- **一度に聞くのは2問まで**（AskUserQuestion）。質問攻めにしない。
- **未定はこちらが複数パターンを出して選ばせる**（ユーザーに考えさせない）。
- **学習ループは常時オン**: 会話に好み・要望（「〜したいって言ってた」「辛いの苦手」）が出たら、その都度 trip-profile モードB で該当者の観察ログへ追記する。
- 「相談だけ」のときは無理にフェーズを進めず、まず雑談的に応じてから「よかったら計画にしましょうか？」と誘う。

## 入力モード（全フェーズ共通・各フェーズ開始時に確認）
各フェーズの冒頭で、ユーザーに進め方を選ばせる。AskUserQuestion で：
- **🙋 自分で決める** — ユーザーが内容を指定し、こちらは整形・裏取り（移動時間/営業時間/相場）・リンク生成などの**補助に徹する**。提案で上書きしない。
- **✨ おすすめにまかせる** — プロファイル＋リサーチでこちらが具体案を出し、ユーザーは**選ぶ/微調整するだけ**。
- **🤝 一緒に決める（既定）** — 叩き台をこちらが出し、対話で詰める。
**項目ごとに混在OK**（例: 行き先は自分で、宿はおすすめ、食はおまかせ）。一度選んでも途中で切り替えてよい。
選んだモードはそのフェーズの進め方に反映する（「自分で決める」を選んだ項目では候補を押し付けない）。

## 個人情報の保護（命名とファイル作成の絶対原則）
公開リポジトリ内で計画しても個人情報が漏れないよう、ファイルの作り方を固定する。
- 個人情報を含むファイル（プロファイル / `wishlist.md` / `trip-basics.md` / `combinations.md` / `plans/` / `output/`）は、**必ず規定の固定名**で作る。これらは `.gitignore` 済みで公開されない。
- **ユーザーに新規ファイルを命名・作成させない**。「自分で書きたい」場合も、**こちら（スキル）が規定名の空ファイル/テンプレを先に作成**し、ユーザーはそこへ追記するだけにする。命名規則を外れて gitignore を擦り抜ける事故を防ぐため。
- 実データを `_example` 付きの名前で保存しない（`_example` は公開される見本専用）。
- リポジトリ内で実際の旅行を計画する場合、作業ファイルはリポジトリ直下（gitignore対象の規定名）に作ってよい。別の旅行を増やすときは `trips/<旅行名>/`（gitignore済み）以下に作る。

## 使い方の例
- ユーザー:「来月ヨーロッパに親友と6泊で行きたいな」
  → フォルダ `~/旅行/<旅行名>/` 作成提案 → 同行者のプロファイル有無を確認 → trip-basics で日程・到着帰宅を詰める → そのまま wishlist 収集 → 案出し … と自然に進む。
- ユーザー:「この前の旅行、しおりだけ作って」
  → 既存フォルダを検出 → 確定 plan を確認 → trip-brochure へ直行。
